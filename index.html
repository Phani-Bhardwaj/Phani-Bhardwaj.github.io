WHY CHECKOV + LLM, NOT LLM-ONLY

Checkov catches known misconfiguration patterns deterministically — zero hallucination risk. The LLM pass catches what static rules can't: setting combinations that add up to a privilege-escalation path even when no single rule fires.

WHY RAG, NOT DIRECT LLM CALLS

Ask an LLM "why is this insecure?" with no retrieval and it answers from parametric memory — which drifts and can't be audited. Every remediation here cites a specific knowledge-base entry, so it's checkable, not just trusted.

Why this, not an existing tool

Checkov, tfsec, and kube-score alone are deterministic and fast — but can't reason about combinations of settings that only add up to a real risk in context. An LLM alone (e.g. pasting a Terraform file into ChatGPT) can reason about combinations, but hallucinates benchmark details and can't be audited. This project refuses to pick one: Checkov stays for what determinism does best, the LLM layer is reserved for judgment calls static rules can't make, and every LLM claim is grounded back to a cited knowledge-base entry instead of trusted blindly.

Scaling to multiple Terraform modules & multiple K8s clusters

As built, the pipeline proves the architecture on a single-module, single-cluster scan — the right scope for a portfolio project without overengineering. The path to production multi-module, multi-cluster scale is concrete, not theoretical:

  • Per-environment knowledge partitioning — scope the RAG knowledge base per cluster/environment instead of one global store, so dev/staging/prod each retrieve against the right baseline.
  • Cross-module dependency resolution — resolve terraform_remote_state / module source references across the whole repo graph before scanning, since the real privilege-escalation path often lives between modules, not inside one.
  • Fan-out instead of one-shot — use LangGraph's parallel Send API to run parser/scanner agents per-cluster concurrently, with the report agent as a single fan-in node aggregating across all of them.
  • Human-in-the-loop before any write — gate remediation writes behind a LangGraph interrupt/resume step once a "fix" means touching infra serving real traffic across clusters, not a sandboxed test config.
